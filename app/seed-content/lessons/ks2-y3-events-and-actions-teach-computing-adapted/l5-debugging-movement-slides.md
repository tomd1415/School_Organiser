# Debugging movement

## Today we are learning
- test a program against a design
- match a piece of code to what it does
- find and fix a bug

> 🧑‍🏫 Read the three "I can…" aloud. Vocabulary on the board: bug · debug · error · design · test. Recap: a bug is an error in a program. Debugging means finding and fixing it. Same routine today.

## The Scratch screen
![A Scratch project: green flag, stop button, a maze on the stage, and an instructions panel]({{res:l5-scratch-screen.png}})

- green flag = start
- red button = stop
- the maze is on the stage

> 🧑‍🏫 Starter. Name the parts of the screen calmly. This is the label task on the starter worksheet.

## Test against the design
- the design says what should happen
- run the program and watch what does happen
- if they do not match, there is a bug

> 🧑‍🏫 I-do. Explain: a bug is when the program does not match the design. We compare "should happen" with "does happen."

## Find the bug
![A block stack: when left arrow key pressed, point in direction 90, move 10 steps]({{res:l5-buggy-code.png}})

- design: left arrow should move **left**
- this code points in direction 90, so the sprite moves **right**
- that is the bug

> 🧑‍🏫 We-do. Walk through it: left arrow but the sprite goes right. The bug is in "point in direction 90". This is the activity worksheet. Likely error: pupils blame the event block → fix-words "the event is fine; the action has the wrong number."

## Fix it
- change point in direction to **-90**
- now the left arrow moves the sprite left
- test it again

> 🧑‍🏫 You-do. Support: change the one number as shown. Core: find and fix it in their own project. Challenge: fix a second bug (a key with pen down where it should be pen up). TA: prompt, do not do it for them.

## Show me
Tick your three "I can…". Show me a bug you fixed.

> 🧑‍🏫 Plenary. Each pupil names one bug they found and how they fixed it.
