# Making an instrument

## Today we are learning
- name the objects I need for a project
- turn a task into a design
- write an algorithm and build it as code
- find and fix a bug (debug)

> 🧑‍🏫 Read the four "I can…" aloud. Vocabulary on the board: design · algorithm · bug · debug · object. Same routine as every lesson.

## What do we need?
![The blocks you can choose from]({{res:l6-motion-block-bank-teach-computing.png}})

- one sprite for each key
- each key plays one note
- give each key a useful name (Note A, Note B …)

> 🧑‍🏫 Starter (10 min). Steer away from "just add the keyboard" — you cannot program one big sprite key by key. On the worksheet pupils fill the blanks and choose names.

## Task → design → code → run
- Task: what is needed?
- Design: what should it do?
- Code: how is it done?
- Run: what does it do?

> 🧑‍🏫 I-do → we-do. An algorithm is a precise set of ordered steps. Plan one key: an event, a sound, a look change.

## Build and debug
![A bug — a mistake in the code]({{res:l6-a-bug-teach-computing.png}})

- build one key, then test it
- a bug is a mistake in the code
- debug = find it and fix it
- when it works, copy the code to the other keys

> 🧑‍🏫 You-do (about 26 min). Likely error: code that does the wrong thing. Fix-words: "a bug is a mistake in the code; debug = find it and fix it." TA: prompt, do not do it for them. Screenshot the finished instrument.

## Showcase
- play your instrument to a partner
- listen to theirs

> 🧑‍🏫 Plenary. Calm sharing. Recap the unit journey on the board.

## I can… and the unit so far
- sprite → sequence → events → sound → design
- you built a working instrument!

> 🧑‍🏫 Tick the four "I can…". Recap: sprite, sequence, events, sound blocks, design and debug — the whole unit in one line.
