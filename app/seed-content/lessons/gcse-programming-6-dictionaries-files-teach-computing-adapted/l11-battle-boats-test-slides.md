# Battle Boats — testing

## Today we are learning
- describe normal, boundary, and erroneous test data
- write a test plan from my success criteria
- record expected vs actual output
- find and fix a bug

> 🧑‍🏫 Read the four "I can…" aloud. Vocabulary on the board: normal · boundary · erroneous · test plan. Penultimate lesson of the project. Calm, steady pace.

## Starter — three types of test data
Write the definitions of **normal**, **boundary**, and **erroneous** test data.

> 🧑‍🏫 Mini-whiteboards. Confirm: normal = expected, boundary = the edge values, erroneous = should be rejected. Grid is 0–7, so 0 and 7 are boundaries, 9 or "cat" are erroneous. Support: match the type.

## A test plan  (I do)
- One row per test: **description · input · expected · actual**.
- Base it on your **success criteria** — test everything you said the program must do.
- Use all three data types where they apply.

> 🧑‍🏫 I-do. Model one test row on the board (e.g. firing at a repeat square). If a test fails, work out the fix and record it. This is the "Testing" row of the rubric.

## Test as a player would  (we do)
- Place a boat on an empty square (normal).
- Fire at row 0 and row 7 (boundary).
- Type 9 or "cat" (erroneous) — the program should ask again, not crash.

> 🧑‍🏫 We-do. Try a boundary and an erroneous input together. Robustness matters for the rubric — does it catch human error?

## Your turn  (you do)
- Complete the testing table from your success criteria.
- Core: write a full test (input, expected, actual) for one criterion.
- Challenge: record a bug and exactly how you fixed it.

> 🧑‍🏫 You-do — most of the lesson. Remind them to test the three data types. TA: prompt, don't test it for them. Screenshot the testing table at the end.

## I can…
Tick your four "I can…". Show me your testing table.
