# Initiate a project

## Today we are learning
- analyse a project brief
- find the user requirements
- spot the constraints and risks
- what a feasibility report is for

> 🧑‍🏫 Read the four "I can…" aloud. Vocabulary on the board: initiate · user requirements · constraints · mitigate · feasibility report. Same routine as every lesson.

## The project life cycle
- **Initiate → Plan → Execute → Evaluate**
- every project follows these stages

> 🧑‍🏫 The source has a life-cycle wheel as a drawn shape — sketch it on the board. Lessons 2–7 work through these stages with one project.

## Meet the project — Delicious Desserts
![Delicious Desserts]({{res:l2-delicious-desserts-logo-teach-computing.png}})

- a dessert shop wants to grow by selling online
- they need an integrated system, a dessert selector, and two posters

> 🧑‍🏫 This brief is used right through to lesson 7. Hand out the brief.

## Analyse the brief
- read it carefully
- highlight or list what is being asked for
- this is the FIRST thing any project needs

> 🧑‍🏫 Model highlighting one paragraph, then writing it as a list.

## User requirements
- a user requirement is what the **client wants** you to do
- e.g. the online system, the dessert selector, the posters

> 🧑‍🏫 Likely error: mixing up a requirement (what they want) with a constraint (what could go wrong). Fix-words: "requirement = want; constraint = risk."

## Constraints and risk
- a constraint could stop the project succeeding — only 8 weeks; the team has basic IT skills
- **mitigate** = plan a way to reduce the risk

> 🧑‍🏫 Use a risk table: the risk, how likely, and how to mitigate it.

## Feasibility report
- a short report that says whether the project can be done
- introduces the company, lists the user requirements, lists the risks

> 🧑‍🏫 Pupils start their feasibility report and screenshot it onto the activity worksheet. TA: prompt, do not do it for them.
