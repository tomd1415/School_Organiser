# Design to code

## Today we are learning
- add the artwork to a game from a design
- choose a variable name that says what it holds
- build a game and test it (debug)

> 🧑‍🏫 Read the four "I can…" aloud. Words on the board: design · code · variable · name · test · debug. Recap: you planned your game last lesson.

## Starter — a chatbot's variables
![Chatbot code with variable_a and variable_b]({{res:l5-chatbot-bad-names.png}})

- the chatbot remembers your **name** and **age** in variables
- but they are called variable_a and variable_b
- those names do not say what they hold

> 🧑‍🏫 Pupils tick where variables are used on the starter worksheet. Support choose why variable_a is a poor name.

## Good names help
![The same chatbot with variables named name and age]({{res:l5-chatbot-good-names.png}})

- rename variable_b → **name**
- rename variable_a → **age**
- now anyone can read the code

> 🧑‍🏫 Likely error: "the name does not matter." Fix-words: "a good name SAYS what it holds." Pupils match the names on the activity worksheet.

## Build from your design
- open your Lesson 4 design
- add your two sprites and your background
- add your variables with good names

> 🧑‍🏫 Pupils order a small block sequence (ask → set → say), then build their own game. TA: prompt, do not build it for them.

## Test and debug
- build your game, then **run** it with the green flag
- **test**: does it work like your design?
- if not, **debug**: find the problem and fix it

> 🧑‍🏫 Games design is a cycle: build, run, test, fix. Pupils order the steps, then screenshot their running game. Challenge: what would you check if the score did not go up?
