fragments, 212 mutating actions). Generated from the route files (`app/src/routes/*.ts`, `app/src/auth/routes.ts`)

context is in [DEVELOPER_INPUTS_RESPONSE.md §2](DEVELOPER_INPUTS_RESPONSE.md); this is the literal list.

**How to read it**

- **Gate** — the route's `preHandler`:
  - `auth` = `requireAuth` (teacher session required);
  - `auth+csrf` = `[requireAuth, app.csrfProtection]` (the standard mutating-route guard) or the inline equivalent;
  - `csrf` = CSRF only (pupil/TA actions authorised separately in-handler via `actingPupil`/session role);
  - `—` = unauthenticated (login, onboarding, health, pupil-public landing).
- **Type** — `page` (renders the `layout()`/`pupilLayout()`/`taLayout()` shell), `fragment` (GET returning an HTMX partial), `action` (POST mutation; may return a fragment or a redirect).
- Server-side enforcement is authoritative; the UI omits/disables affordances only. Feature gates that
  sit *inside* a handler (e.g. `marksEnabled()`, `pupilAccessEnabled()`) aren't in the Gate column — see
  the permission matrix in the response doc (§5) and the per-page handoffs ([PAGE_HANDOFFS.md](PAGE_HANDOFFS.md)).
