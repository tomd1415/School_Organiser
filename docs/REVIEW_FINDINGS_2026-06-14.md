### 1. Multi-word pupil names with non-canonical internal whitespace bypass BOTH redaction and the egress assert — name reaches the AI provider
- **Where:** `app/src/services/redact.ts:36-44 (redactNames) and :55-61 (containsRosterName)`  ·  **lens:** ai-privacy
- **What/why:** redactNames/containsRosterName build the match regex from the stored display_name verbatim via escapeRegExp(name), with only NFC normalisation applied. NFC fixes accented letters but does NOT collapse whitespace runs. A roster name stored as "Anna Lee" (single ASCII space) therefore fails to match the same name written with a non-breaking space (U+00A0), a tab, or two spaces. Verified directly against the real functions: for roster [{displayName:'Anna Lee', aiToken:'PUPIL_1'}], redactNames('Anna Lee was here') returns 'Anna Lee was here' (NOT redacted) and containsRosterName(...) returns false (egress assert does NOT block). Same for a tab ('Anna\tLee') and a double space ('Anna  Lee'). Because the assert also misses it, the unredacted full name is then sent verbatim in messages.create / messages.parse (client.ts:158-163, 206-211). Non-breaking spaces, tabs and double spaces are routine in pasted emails, MIS/SIMS CSV exports, and pupil-typed worksheet answers (markOpen sends raw a.value through markAnswersItems → context → redactNames). This is a direct breach of the project's #1 non-negotiable ('No pupil name is EVER sent to any AI service'). Single-token names ('Sam') are unaffected; only multi-word names — i.e. most real SEND rosters.
- **Repro:** 1) Add a pupil 'Anna Lee' on /pupils. 2) In a pupil worksheet answer (or a forwarded email body, or a captured note routed to AI), include the text 'Anna Lee' (an nbsp between the names — produced by copy-paste from Word/Outlook/PDF) or 'Anna  Lee' (double space). 3) Trigger any AI feature whose context carries that text (auto-mark open answers, email triage, smart-file note, class-work summary). The full name 'Anna Lee' egresses to Anthropic, and the ai_calls audit row stores it too — both contrary to the privacy guarantee.
- **Fix:** Normalise internal whitespace on BOTH sides before matching: collapse runs of any Unicode whitespace to a single space in the input text and in the stored name, and/or build the name regex so each inter-token space matches \s+. Concretely, in nameRegExp split the (NFC-normalised, escaped) name on /\s+/ and join the parts with '\\s+', and pre-normalise the haystack text the same way in both redactNames and containsRosterName. Add tests for nbsp/tab/double-space variants of a two-word name.

- **What/why:** The wizard renders already-moved source groups as read-only 'done' (rollover.ts:54-56, driven by listRolloverGroups' successorId). But the POST handler does not re-check whether a source group already has a successor in the target year — it only loops over the ticked `take[]` ids and calls rolloverGroup, which blocks solely on a duplicate NAME in the target year (setup.ts:384-391). So if the teacher submits the form again for an already-moved group under a DIFFERENT name (e.g. tweaks the suggested name, or replays a stale form), a second successor group is created for the same predecessor_group_id, splitting that class's continuity/history chain into two. The 'idempotent and re-enterable' guarantee in the file header (rollover.ts:1-3) holds only for identical names.
- **Repro:** Move 7ARO up to '8ARO'. Re-open the wizard with a stale/back-button form (or POST directly) ticking 7ARO with name '8ARO-new'. rolloverGroup sees no name clash and inserts a second group with predecessor_group_id = 7ARO, giving the source class two successors.
- **Fix:** In rolloverGroup (or the POST handler), skip/refuse when the source group already has a successor in the target year: add `AND NOT EXISTS (SELECT 1 FROM groups WHERE predecessor_group_id = $sourceId AND academic_year_id = $targetYear)` before inserting, returning null so the wizard reports it as already moved.
